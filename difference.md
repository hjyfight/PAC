## 二、三类差异,分开看

### 1. 绝对 mAP50 不可直接对比 —— 且原因要修正(不是"低阈值假阳性洪水")

复现 Gray=20.5、R≈42,光看 mAP 像"强攻击",但检出率 98–100%,根本没藏住人。之前把这归因为"conf=0.01 低阈值下 patch 锐利边缘制造假阳性洪水"——**这个解释已被 conf=0.25 的复跑推翻**:`output_new/table1_yolov5s_conf25.json` 里 Gray mAP=15.5,反而更低(clean 仍 94.2)。

真实原因:在我们的放置协议下(每个 GT bbox 中心贴 patch、占 bbox 面积 25%),任何遮挡(哪怕灰块)都会让人体框**碎裂/偏移**,产生大量**高置信度但 IoU<0.5** 的框,precision 被砸穿;提高阈值救不回。论文的放置/匹配协议必有不同(论文未给细节),所以绝对 mAP 永远对不上。

👉 结论照旧、理由更新:**按检出率(或最大人体置信度)排名,不要看绝对 mAP50**。写论文解释 mAP 假象时用"遮挡致框错位/碎裂(高置信度假阳性)",不要写"低置信度假阳性洪水"。

### 2. 真实复现成功的部分(检出率口径)

- **AdvPatch**:mAP 31.7 vs 论文 31.6,检出率 39.6% —— 几乎完美复现,仍是最强攻击。
- **CAPGen-T1/T2**:检出率 61%,平均最大置信度 0.854→0.54。比论文(71.5/71.7,中等攻击)更强,方向完全正确;偏强可能因为贴所有 bbox、单检测器白盒、patch 尺寸算法差异。T1≈T2(两组底色等效)与论文一致。
- **Gray / Random 无效**:与论文定性一致。R 已改为论文 Eq.5 的连续随机权重(旧版是随机 one-hot 硬分配),结论不变。

### 3. 真正没复现的点:CAPGen-P(论文招牌),且证据已闭环

论文核心卖点:P1=32.7 ≈ AdvPatch 31.6 ≪ T 71.5,据此论证"pattern 比 color 更重要、换色保攻击"。我们用**三种构造**都复现不出来,而且最后一种已排除"实现破坏 pattern"的嫌疑:

| 构造 | P_orig(换色前) | P1 / P2 | 死因 |
|---|---|---|---|
| ① K-means 硬量化 AdvPatch(旧版,2026-06-07) | det 0.86 | 0.93 / 0.95 | 硬量化先毁掉幅值结构 |
| ② soft 最优分解 AdvPatch(`table1_yolov5s_softp_with_t.json`) | det 0.608(AdvPatch 本体 0.431) | 0.93 / 0.91 | 数学上不可能:3 色凸组合只张成 RGB 立方体中的 2D 三角形,free-pixel patch 重建只有 PSNR≈13.3 dB,换色前攻击已丢 |
| ③ **faithful Eq.4:训练好的 3 色 pattern 直接换底色(当前默认,本表)** | **det 0.552(P0===源,max\|diff\|≈2e-7)** | **0.799 / 0.920** | **换色这一步本身就毁攻击** |

③ 的关键:P0 与源 patch 逐像素一致,P1/P2 的衰减 **100% 来自 Eq.4 换色**。即使给论文方法最有利的构造,"换色保攻击"也不成立;而 T1(0.611)证明同样的 Bc1 直接训练就有效 —— 与论文 P≪T 的排名**相反**。

**机理(为什么换 Bc1/Bc2 会毁)——亮度动态范围被压缩,且无法靠重排颜色修复:**

- 源底色(INRIA 环境 K-means)≈ 三档灰:luma ≈ 47 / 115 / 189,跨度 142;
- Bc1 luma ≈ {62, 73, 120}:跨度压到 58;Bc2 luma ≈ {59, 87, 109}:跨度 50。任何颜色对应下逐色亮度距离 lumD ≥ 120,动态范围只剩源的 35–40%。

与 `output_new/capgen_t/pattern_vs_color.json`(旧 capgen_t 上测得,方向通用)一致:逐色保亮度的换色不掉攻击(纯灰度 0.535、hueA 0.629 ≈ 源 0.556),亮度失真大的 palette 攻击弱(snow 0.938 最差)。

**排列-recolor 实验(2026-06-11,`run_perm_recolor.py` → `output_new/perm_recolor_yolov5s.json`)把"颜色对应错了"这条质疑也堵死了**。Eq.4 未规定新旧颜色对应顺序,于是把 Bc1/Bc2 各 3!=6 个排列全部评测(协议与终表逐字节一致;锚点 P0 / Bc1-012==P1 / Bc2-012==P2 与终表三位小数全对上):

- **12 个排列 detRate 全部落在 0.799–0.934**,最好的(Bc1-012,0.799,恰为论文的恒等对应)也远弱于 P0 0.552 和 T1 0.611 → **不存在能保攻击的颜色对应,recolor ≠ retrain**;
- 机理表述修正:亮度**次序**匹配的排列(Bc1-021 0.854、Bc2-021 0.934)并**不**更强,palette 内排序与次序/lumD 皆不单调(lumD=243 的 Bc1-012 反而最强);两色集 6 排列均值几乎相同(0.872 vs 0.875),所以"Bc2 跨度更窄→P2 更弱"也不成立——恒等对应下 P1/P2 的差距(0.799 vs 0.920)主要是对应运气。**一阶因素是跨 palette 的亮度动态范围/逐色亮度保真;palette 内的具体对应只是二阶效应**(极差 0.135≈39/288 张,相邻名次 ±0.03≈±9 张,接近噪声),写论文用"范围压缩",不要用"次序颠倒"。

## 三、"pattern > color" 这个结论到底复现了没有?

以忠实口径,**部分成立,但必须重新表述**:起作用的是"相对像素幅值/亮度结构",不是色相。

- ✅ "hue 不重要"成立:纯灰度(完全去色)攻击不掉(0.535 ≈ 0.556);
- ❌ "随便换成环境基础色都保攻击"(论文 Table 1 的 P 行)不成立:新底色必须**逐色保持亮度(尤其是动态范围)**,Bc1/Bc2 在全部 12 种颜色对应下都失败(detRate ≥ 0.799)。

**一句话总结给你写论文用:** 训练管线、AdvPatch、CAPGen-T 复现成功且方向正确(AdvPatch 31.7 vs 31.6 几乎复刻);绝对 mAP 因放置/匹配协议不可对标(遮挡致高置信度框碎裂,调阈值无效),应改用检出率;论文招牌 CAPGen-P 用三种构造(含严格 Eq.4 精确换色)都复现失败、且 P/T 排名反转,且 Bc1/Bc2 的全部 12 种颜色对应都试过(最好 detRate 0.799,均失败),机理是 Bc1/Bc2 把源 pattern 的亮度动态范围压缩到 ~40%;"pattern>color"仅在"亮度结构 > 色相"的意义上成立。

## 四、收尾实验

1. ✅ **排列-recolor(已完成,2026-06-11)**:Bc1/Bc2 全部 6 排列 × 2 色集,结果见二、3 机理小节。结论:无任何颜色对应能保攻击(最好 Bc1-012=0.799,远弱于 P0 0.552 / T1 0.611),"换色对应错了"的质疑已堵死;同时机理表述修正为"动态范围压缩是一阶因素,次序不是"(原预期"对齐排列最强"被证伪)。产物:`output_new/perm_recolor_yolov5s.json`、`output_new/capgen_p_perm/`(13 套 .pt+PNG)。
2. ⬜ **在重训后的 capgen_t 上重跑 `test_pattern_vs_color.py`**:现引用的 pattern_vs_color.json 是 EOT 修正前的源 patch 测的;重跑一遍让机理证据与终表同一套 checkpoint。

## 附:历史分析(2026-06-07,K-means 版;已按其结论重写 make_capgen_p.py,仅留档)

> 状态:下述批评针对旧版 K-means 硬量化实现,已被采纳——`make_capgen_p.py` 现默认 `--method recolor-trained`(忠实 Eq.4),`soft`/`legacy-kmeans` 仅作"free-pixel 分解不可行"的批判性对照保留。文末提出的验收标准("P_orig 换色前必须 ≈ 源攻击力")在 recolor-trained 下已按构造满足(P0===源);而 soft 分解证明对 free-pixel AdvPatch 该标准**数学上不可达**(PSNR 13.3 dB)。

论文流程大概是:
先训练一个颜色不受限制的 AdvPatch。把这个 AdvPatch 看成两部分:pattern = 相对像素幅值/纹理结构;color = 基础颜色集合。生成 CAPGen-P1/P2 时不重新训练 pattern,只把颜色替换成两个指定基础色集合:

- Bc1 = [[119,49,72], [2,204,1], [134,2,182]]
- Bc2 = [[199,21,131], [40,165,4], [16,69,120]]

公式是论文 Eq.4:t_ij = Σ_k nc_k · r_k,其中 r_k 是保留的 pattern/color allocation,nc_k 是新基础色。即论文的 CAPGen-P 是"强攻击 pattern + 新环境 base colors"的换色,不重新训练。

旧实现的最大差异:旧版 `make_capgen_p.py` 把 AdvPatch 的 RGB 像素做 KMeans(n_clusters=3) 硬 one-hot 量化。论文没有这么说过(论文的 K-means 只用于从环境图提取 base colors),这个操作破坏了 AdvPatch 的连续明暗、细小扰动和相对幅值结构——证据:量化后用原聚类中心渲染(P_orig)攻击已明显变弱,所以问题不在 Bc1/Bc2 选色,而在 pattern 提取阶段已损坏。

| 部分 | 论文 CAPGen-P | 旧复现(K-means) | 现复现(recolor-trained) |
|---|---|---|---|
| 来源 | 先训练强 patch | AdvPatch | 训练好的 3 色 CAPGen-T |
| Pattern | 保留相对幅值结构 | K-means 硬量化(损坏) | 原样保留(精确) |
| Color | 换成 Bc1/Bc2 | 同 | 同 |
| 重新训练 | 否 | 否 | 否 |
| P_orig≈源? | (未说明) | 否(0.86 vs 0.43) | **是(===,0.552)** |

最终判断:论文的 CAPGen-P 是"固定强攻击 pattern 的换色方法",但论文未给出 free-pixel patch → r_k 的可行分解算法;我们证明该分解数学上不可行(3 色=2D 平面),而绕开分解、用本就有 (r_k, colors) 结构的训练 pattern 做精确换色后,**换色本身仍然毁攻击** —— 论文 Table 1 的 P 行在任何构造下都未能复现。
